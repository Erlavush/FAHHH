---

## Visual Direction

Use a warm, sunlit "keepsake apartment" player shell and a separate utilitarian workshop developer shell.

- Player view is room-first. Chrome should feel like pinned cards, glass, and paper around the room edges, not a debug HUD.
- Developer view is tool-first. Use darker neutral panels, tighter spacing, mono labels, and explicit section headers.
- Never reuse developer panel styling in the player shell.
- Never show technical identifiers, debug toggles, or authoring entry points in the default player HUD.
- Player copy uses sentence case. Developer workspace labels may use uppercase section headers.

---

## View Split Contract

| Area | Contract |
|------|----------|
| Default shipped mode | `Player View` only |
| Default dev mode | Remember last selected mode. First-time dev default may be `Developer View` for iteration speed |
| View switch visibility | Dev-only segmented switch. Never render in shipped player UI |
| Player View purpose | Real shipped experience for room play, build mode, inventory, progression, partner status, settings, and PC minigame |
| Developer View purpose | Fast iteration workspace for Dev Panel, world controls, preview settings, Preview Studio, furniture preview, Mob Lab, shared-session diagnostics, and performance tooling |
| State handoff | Switching views must not mutate room state, build state, inventory state, preview selections, or shared-session state |
| Authoring boundary | Preview Studio and Mob Lab remain explicit authoring tools and never appear automatically in Player View |

## Player View Layout

| Zone | Contents | Rules |
|------|----------|-------|
| Top-left | Read-only room clock chip | Time only. No settings, no build toggles, no debug labels |
| Top-right | Companion card plus settings button | Companion card is collapsed by default and expands for partner or ritual detail |
| Bottom-left | Progress stack | Wallet, level, XP, and streak. Max 2 compact cards visible at once |
| Bottom-center | Primary action dock | `Enter Build Mode`, `Open Inventory`, contextual interaction action, current mode state |
| Left drawer | Inventory and build catalog | Opens from Player View only when explicitly requested. Max width 400px desktop |
| In-scene overlays | Existing edit dock, placement actions, interaction hints | Keep close to the selected object; do not move object editing into a corner panel |
| Full-screen overlays | Entry shell, blocking overlay, PC minigame | One modal layer at a time |

### Player View rules

- Do not stack more than one persistent panel in the same corner.
- Keep persistent HUD height under 72px per edge zone.
- Hide room ID by default in Player View.
- Only show invite code when the partner is missing and the companion card is expanded.
- Move `Reload latest room` out of always-visible HUD chrome. Put it inside expanded room details as a secondary action.
- Move `Import Minecraft Skin` into Settings or Profile, not the primary action dock.
- Keep build mode and inventory in Player View, but move `Grid Snap` into a secondary build settings sheet instead of a top-level button.

## Developer View Layout

| Zone | Contents | Rules |
|------|----------|-------|
| Top header | View switch, room/session badge, quick open actions | Dev-only shell chrome |
| Left rail | Workspace navigation | Tabs: `Room`, `Inventory`, `Preview Studio`, `Mob Lab`, `World`, `Session` |
| Center stage | Room runtime or authoring stage | Room View remains the default stage; Preview Studio and Mob Lab replace the stage inside Developer View |
| Right inspector | Dev Panel and selected-context details | Reuse current Dev Panel sections with stronger grouping and persistence |
| Bottom utility area | Performance readout or temporary logs if needed | Optional; never required for Player View |

### Developer View rules

- `Preview Studio`, `Furniture Studio`, and `Mob Lab` open inside Developer View, not as floating player modals.
- Keep world clock controls, lighting controls, collision toggles, and reset actions in Developer View only.
- Keep `Reset Sandbox`, room reload/debug actions, room IDs, invite codes, and performance data in Developer View only.
- Preserve fast access to current dev flows with one click from the rail or header.
- Use darker neutral surfaces and mono utility text so the workspace reads as tooling, not gameplay.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| View switching | Instant, no reload, preserves open drawer/tab state per view |
| Companion card | Collapsed by default. Expands on click/tap. Expanded state may reveal invite code, ritual explanation, and manual room refresh |
| Build mode entry | Triggered from bottom-center dock. Entering build mode may auto-open the inventory drawer if it was previously open |
| Inventory | Opens as a dedicated drawer, not a floating panel over other HUD cards |
| Grid snap | Secondary toggle inside build settings, not primary dock chrome |
| Settings | Single settings trigger in top-right. Player-safe items only |
| Preview Studio and Mob Lab | Developer View only. Opening either replaces the center stage or opens a dedicated workspace pane |
| PC minigame | Full-screen modal with blurred backdrop. Closing returns to prior player room state |
| Manual room refresh | Secondary action only. If uncommitted edits exist, require confirmation before replacing local view state |

---

## Responsive Contract

- At `<= 900px`, collapse the Player companion card and settings into one top-right sheet.
- At `<= 900px`, convert the inventory drawer into a bottom sheet or near-full-width drawer. Only one drawer or modal may be open at a time.
- At `<= 900px`, allow the bottom action dock to wrap to two rows, but keep it centered and readable.
- At `<= 1200px`, Developer View may collapse the right inspector behind a drawer toggle, but the left rail and stage must remain usable.
- Do not let Player View inherit Developer View sidebars on mobile.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Enter Build Mode |
| Empty state heading | Your partner is not here yet |
| Empty state body | Keep decorating solo for now, or open Room details to copy the invite code and bring them in. |
| Error state | Room sync fell behind. Refresh the latest room state and try the action again. |
| Destructive confirmation | Sell furniture: `Sell {item_name} for {coin_amount} coins?` |
| Destructive confirmation | Refresh room state while editing: `Discard your unconfirmed changes and load the latest shared room?` |
| Destructive confirmation | Developer reset: `Reset the local sandbox and developer workspace state? This cannot be undone.` |

### Status copy rules

| State | Title | Body |
|-------|-------|------|
| Partner missing | Waiting for partner | Send the invite code from Room details when you are ready to decorate together. |
| Partner connected | Together now | Both partners are in the room and the latest shared state is loaded. |
| Partner reconnecting | Reconnecting | Hold tight while the room catches up to your partner's latest presence. |
| Ritual pending | Daily ritual pending | Both partners still need one desk check-in today. |
| Ritual complete | Daily ritual complete | Today's shared check-in is done. Come back tomorrow to protect the streak. |

## Component Notes

- Preserve the existing in-scene edit dock and floating placement actions because they already keep editing close to the object.
- Replace all-caps player buttons with sentence case labels.
- Keep `Shared Inventory` language in the drawer, but shorten helper copy and remove debug phrasing.
- Replace always-visible "Reload latest room" with a quieter secondary action label: `Refresh room state`.
- Keep `Pixel Gigs` as the PC minigame title in Player View.
