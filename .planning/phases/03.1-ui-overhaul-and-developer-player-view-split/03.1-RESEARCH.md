## User Constraints (from user request + UI-SPEC.md)

### Locked Decisions

### View Split
- **D-01:** The shell must split into explicit `Player View` and `Developer View` surfaces.
- **D-02:** `Player View` is the shipped experience and must feel aesthetic, structured, and room-first.
- **D-03:** `Developer View` must keep fast access to Dev Panel, Preview Studio, furniture preview flows, Mob Lab, world controls, and debug/session tools.
- **D-04:** View switching must not mutate room state, build state, inventory state, preview selections, or shared-room session state.

### Brownfield Preservation
- **D-05:** Preserve the existing shared-room runtime, build mode, inventory flow, settings flow, and preview flows.
- **D-06:** Preserve Preview Studio and Mob Lab as explicit authoring tools and keep them out of the default player shell.
- **D-07:** Preserve the in-scene edit dock and placement action overlays.
- **D-08:** Keep `RoomView` as the live runtime owner and avoid introducing a second gameplay runtime path.

### Player Shell Contract
- **D-09:** Player chrome must stop clustering time, status, build/inventory, and utility actions into the same top-left overlay stack.
- **D-10:** The player shell must follow `03.1-UI-SPEC.md`: warm parchment dominant surfaces, deep ink secondary surfaces, terracotta accent, room-first composition.
- **D-11:** `Reload latest room` becomes a secondary room-details action, not persistent HUD chrome.
- **D-12:** `Import Minecraft Skin` and `Grid Snap` move out of primary player controls and into secondary settings/details surfaces.

### Developer Workspace Contract
- **D-13:** Developer workspace uses darker neutral tooling surfaces, mono utility text, and explicit navigation for `Room`, `Inventory`, `Preview Studio`, `Mob Lab`, `World`, and `Session`.
- **D-14:** Performance readouts, room IDs, invite codes, and raw shared-session detail are developer-only.
- **D-15:** Preview Studio and Mob Lab should run inside the developer workspace instead of floating over the player shell.

- Exact persisted shell preferences inside `devWorldSettings.ts`.
- Exact implementation of room-details/settings sheets inside the player shell.
- Exact internal naming for new player-shell and developer-workspace components.

### Deferred Ideas (OUT OF SCOPE)

- shadcn/Tailwind/design-system migration.
- Replacing `RoomView` or the existing room editor interaction model.

## Summary

The current problem is structural, not cosmetic. `src/App.tsx` still renders the player shell, shared-room status strip, dev panel, inventory panel, preview studio modal, performance monitor, and PC minigame from one absolute-position overlay stack. `src/styles.css` then compensates with sibling-position rules such as `.shared-room-status + .scene-toolbar` and `.shared-room-status ~ .spawn-panel`, which is why every new surface competes for the same space.

Phase 03.1 should start by splitting the shell into two explicit branches driven by one persisted view-mode state in `devWorldSettings.ts`. The correct move is not "restyle SceneToolbar"; it is "stop using SceneToolbar as the app shell." `Player View` should own only room-first HUD surfaces, drawers, and player-safe modal overlays. `Developer View` should become a workspace shell that keeps the same room runtime but reorganizes tooling into a left navigation rail, center stage, and right inspector.

No new UI framework is required. The repo already has the correct primitives: React local state in `App.tsx`, brownfield persistence in `devWorldSettings.ts`, and dedicated tool/runtime components. The planner should favor component extraction and pure view-model helpers over introducing shadcn, Tailwind, routing, or a second global state layer.

**Primary recommendation:** Add a persisted `AppShellViewMode` plus developer workspace tab, extract player-shell and developer-workspace layout components under `src/app/components`, move preview and debug tooling entirely behind the developer branch, and then rebuild the player HUD zones according to `03.1-UI-SPEC.md`.

### Pattern 1: Persisted shell mode and workspace tab in `devWorldSettings.ts`
**What:** Extend the existing world-settings persistence boundary with a shell view mode plus developer tab selection.
**When to use:** For `Player View` vs `Developer View` and current developer workspace tab only.
**Example:**
```typescript
export type AppShellViewMode = "player" | "developer";
export type DeveloperWorkspaceTab =
  | "room"
  | "inventory"
  | "preview_studio"
  | "mob_lab"
  | "world"
  | "session";

export interface PersistedWorldSettings {
  shellViewMode?: AppShellViewMode;
  developerWorkspaceTab?: DeveloperWorkspaceTab;
  playerCompanionCardExpanded?: boolean;
}
```

### Pattern 2: Split `App.tsx` into shell containers, not more overlay conditionals
**What:** Extract two layout components that accept already-owned runtime state from `App.tsx`.
**When to use:** For the main shared-room shell render branch.
**Why:** The current issue is composition. New components should organize existing surfaces, not duplicate game state.

### Pattern 3: Use pure view-model helpers for player and developer copy/state mapping
**What:** Create `src/app/shellViewModel.ts` that converts runtime state into specific HUD strings, chip tones, and action labels.
**When to use:** For companion-card copy, action-dock primary labels, and developer tab metadata.
**Why:** The repo has Vitest but not a full React UI testing stack. Pure helpers make 03.1 behavior testable without adding React Testing Library.

### Pattern 4: Keep room editing controls in-scene
**What:** Preserve `EditDock` and `PlacementActions` near the selected object.
**When to use:** Always. 03.1 should not move object editing into a side panel.
**Why:** The existing in-room editing flow is already spatially correct and outside the scope of this shell split.

### Pattern 5: Preview Studio becomes a developer workspace stage, not a player modal
**What:** Add a workspace presentation mode to `FurniturePreviewStudio.tsx` so it can render inside the developer center stage.
**When to use:** Developer tabs `preview_studio` and `mob_lab`.
**Why:** This keeps preview tooling fast to reach without ever polluting the player shell.

### Recommended Sequencing
1. Add persisted shell mode plus developer tab state and extract player/developer shell containers.
2. Rebuild the player HUD zones around the `UI-SPEC.md` contract and convert inventory/settings into drawers or sheets.
3. Reorganize developer tooling into a workspace rail/header/stage/inspector layout and embed Preview Studio there.

## Common Pitfalls

### Pitfall 1: Restyling the current stack without changing ownership
**What goes wrong:** The app still has one shell, only with prettier buttons.
**How to avoid:** Establish the `Player View` / `Developer View` branch in `App.tsx` first and move components into those branches before visual redesign.

### Pitfall 2: Duplicating room state per view
**What goes wrong:** Switching views resets build mode, selected preview state, or shared-room information.
**How to avoid:** Keep `App.tsx` as the owner of room/runtime state and pass it into extracted shell components.

### Pitfall 3: Breaking build-mode reachability in Player View
**What goes wrong:** The new player shell looks cleaner but hides inventory, build mode, or settings behind the developer branch.
**How to avoid:** Keep build mode, inventory, settings, and shared-room player-safe details in Player View; only move debug/authoring/session diagnostics to Developer View.

### Pitfall 4: Leaving Preview Studio as a floating player modal
**What goes wrong:** The developer workspace exists, but the heaviest tooling still overlays the player shell.
**How to avoid:** Give `FurniturePreviewStudio.tsx` an embedded presentation path and mount it inside the developer workspace stage.

### Pitfall 5: Preserving CSS offset hacks
**What goes wrong:** New components still inherit the old overlap behavior because `styles.css` keeps sibling-positioning rules.
**How to avoid:** Remove dependence on `.shared-room-status + .scene-toolbar` and related sibling selectors; replace them with explicit shell layout classes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| View-specific state stores | A second global UI store | Existing `App.tsx` state plus `devWorldSettings.ts` persistence | The current shell already owns the right state; it only needs better layout boundaries. |
| Developer tooling routing | A new router or page system | One in-app developer workspace branch and tab model | 03.1 is an urgent brownfield shell refactor, not a navigation overhaul. |
| Player settings placement | A new full settings dashboard | A compact player details/settings sheet | Keeps the player shell minimal and room-first. |
| Player edit controls | Side-panel placement editing | Existing `EditDock` and `PlacementActions` overlays | The spatial object-edit workflow already works. |

## Open Questions

1. **Should development builds default to `Developer View` or remember last mode only after first load?**
   - Recommendation: In dev, default to the persisted value if present; otherwise initialize to `developer` once for first-time iteration speed. In non-dev, force `player`.

2. **Should `InventoryPanel` be fully duplicated for player and developer layouts?**
   - Recommendation: Keep one core `InventoryPanel` component and wrap it in different shells (`drawer` vs `workspace panel`) so buy/place/sell logic stays single-sourced.
