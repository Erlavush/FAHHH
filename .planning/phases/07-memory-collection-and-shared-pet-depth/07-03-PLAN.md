# Plan 07-03: Runtime and Store Integration

**Goal:** Integrate the expanded memory and pet roster into the shared-room document, store, and runtime orchestration.

## Context
Phase 07 Plan 03 ties together the schema and logic updates from Plans 01 and 02. It shifts the shared room from a single-pet model to a roster-based model and ensures the commit flow correctly handles the richer personalization state.

## Tasks

1.  **Document & Store Schema Update:**
    *   Update `SharedRoomDocument` in `src/lib/sharedRoomTypes.ts`: replace `sharedPet: SharedRoomPetRecord | null` with `sharedPets: SharedRoomPetRecord[]`.
    *   Update `CommitSharedRoomStateInput` in `src/lib/sharedRoomStore.ts` to reflect the new roster field.

2.  **Runtime & Snapshot Update:**
    *   Update `SharedRoomRuntimeSnapshot` in `src/app/hooks/shared-room-runtime/runtimeTypes.ts`.
    *   Update `createSharedRoomRuntimeSnapshot` in `src/app/hooks/shared-room-runtime/runtimeSnapshot.ts` to include migration logic for legacy `sharedPet` records and the new `collectionId` field in memories.

3.  **Commit Flow Update:**
    *   Update `commitRoomState` and `commitRoomMutation` in `src/app/hooks/shared-room-runtime/roomCommits.ts` to correctly pass the updated roster and memory records.

4.  **Verification:**
    *   Update `tests/sharedRoomRuntime.bootstrap.test.ts` or add a new test to verify legacy document migration and multi-pet roster preservation in the runtime.

## Success Criteria
- [ ] `SharedRoomDocument` supports a `sharedPets` array instead of a single pet.
- [ ] Legacy shared-room documents (with single `sharedPet`) are automatically migrated to the `sharedPets` roster on load.
- [ ] The runtime correctly snapshots and commits the expanded roster and collection-aware memories.
- [ ] Regression tests for the shared-room runtime pass.
