# Milestone v1.0: Shared Room MVP

**Status:** SHIPPED 2026-03-27
**Phases:** 1-4, including inserted Phase 03.1
**Total Plans:** 15

## Overview

This milestone turned the proven solo sandbox into a jam-ready shared couple-room MVP on the current local/dev shared-room stack. It shipped canonical shared-room identity, committed shared edits, live partner presence, personal and couple progression, a player-first room shell split from developer tooling, and the first personalization-and-stakes loop with shared memories, one shared cat, and breakup reset consequences.

## Phases

### Phase 1: Shared Room Backbone

**Goal**: Turn the current solo sandbox into a couple-linked room that can load and persist one authoritative room state without regressing existing room invariants.  
**Depends on**: Nothing (brownfield foundation already exists)  
**Requirements**: `PAIR-01`, `ROOM-01`, `ROOM-02`, `ROOM-03`  
**Plans**: 3 plans

**Success Criteria**
1. Couple can pair into one shared room and both clients load the same committed room contents.
2. Confirmed place, store, sell, and remove actions persist to shared state without inventory drift.
3. Shared room state survives refresh, reconnect, and migration from the current solo model.
4. Existing ownership, placement, and authoring boundaries remain intact while shared persistence is introduced.

Plans:
- [x] `01-01`: Define shared room identity, pairing flow, and migration contract from solo saves
- [x] `01-02`: Add authoritative shared-room persistence adapters around the current room schema
- [x] `01-03`: Sync committed room-edit operations while preserving ownership-versus-placement rules

### Phase 2: Live Presence and Co-op Consistency

**Goal**: Make the shared room feel alive for two people while concurrent actions converge predictably.  
**Depends on**: Phase 1

2. Join, leave, and reconnect state is visible and reliable.

## Milestone Summary

**Decimal Phases**

- Phase `03.1`: UI Overhaul and Developer-Player View Split was inserted after Phase 3 because the mixed player/debug shell had become a direct product and workflow blocker.

**Key Decisions**

- Extend the current room schema and commit only confirmed shared-room mutations instead of reviving deleted backend architecture.
- Keep live presence, soft locks, and other transient collaboration signals outside canonical room revisions.
- Split the shipped player shell from the developer workspace before layering late-MVP personalization and stakes.
- Keep Preview Studio and Mob Lab explicit authoring tools instead of promoting them into default player UI.
- Route breakup reset through one pure canonical mutation helper so destructive state changes keep the same replay/convergence guarantees as other shared-room edits.

**Issues Resolved**

- The solo-only room builder now supports one canonical shared room without losing ownership-versus-placement invariants.
- Partner presence, join/reconnect feedback, and same-item edit convergence now behave predictably in the live room.
- Personal wallets, XP, couple streaks, and the first ritual loop now persist inside the shared-room document.
- The shipped room experience now has a room-first player shell instead of a stacked development toolbar.
- Shared memories, one canonical shared cat, and breakup-reset stakes now complete the MVP emotional loop.

**Issues Deferred**

- Production backend and auth for one-couple room ownership remain unselected.
- Phase verification artifacts are still missing for Phases `2`, `3.1`, and `4`; milestone completion accepted that process debt and archived the audit with gaps.
- The next milestone scope is not defined yet.

**Technical Debt Incurred**

- `src/lib/devLocalState.ts` still rejects `wall_front` and `wall_right` on the legacy sandbox path.
- `node ... gsd-tools.cjs roadmap analyze` still reports missing phase details for `3.1` despite the roadmap containing those details.
- `npm run build` still emits large vendor chunk warnings for the current bundle split.
